# v4_1_fix

Ver.4（[メニュー方式]）で見つかった不具合を修正した派生バージョン。

## 修正内容：タイムスタンプの二重カッコ

### 症状

`diary.txt` から読み込んだ過去の記録を表示すると、日時が以下のように末尾の `]` が二重になって表示される。

```
[2026-07-13（月） 06:07:42]]
```

### 原因

`load_entries()` が `[YYYY-MM-DD（曜） HH:MM:SS]` という行を読み込む際、

```c
line[strcspn(line, "\n")] = '\0';
strncpy(temp.timestamp, line + 1, sizeof(temp.timestamp) - 1); // [ を除く
```

のように、改行を取り除いたあと **先頭の `[` だけ** を除いて `timestamp` に格納していた。末尾の `]` はそのまま残るため、`temp.timestamp` の中身は `2026-07-13（月） 06:07:42]` のように閉じ括弧付きの文字列になる。

これを表示時に

```c
printf("[%s]\n%s\n\n", entries[i].timestamp, entries[i].content);
```

のように改めて `[` `]` で囲むため、閉じ括弧が二重になっていた。

このバグは Ver.3 の `load_entries()` 導入時から存在し、Ver.4 にもそのまま引き継がれていた。

### 修正

改行除去のあとに、末尾が `]` であれば取り除く1行を追加した。

```c
// 改行を除去
line[strcspn(line, "\n")] = '\0';

// ★修正点：末尾の ']' を除去
size_t llen = strlen(line);
if (llen > 0 && line[llen - 1] == ']') {
    line[llen - 1] = '\0';
}

strncpy(temp.timestamp, line + 1, sizeof(temp.timestamp) - 1);
```

### 修正後の表示例

```
【過去2件の記録】
[2026-07-13（月） 06:10:43]
今日は修正のテストをしました。
```

括弧が正しく1つだけになっていることを確認済み。

## それ以外の変更点

メニュー・入力受付・新規保存のロジックはVer.4と同じ。差分は `load_entries()` 内の上記1点のみ。

## コンパイル・実行

```
gcc -o diary_v4_1 diary_v4_1.c
./diary_v4_1
```

## 補足：表示される時刻について

`write_new_entry()` は `time()` と `localtime()` を使ってタイムスタンプを生成しているため、実行環境の **タイムゾーン設定** に依存する。GitHub Codespaces のコンテナは既定でタイムゾーンが UTC（協定世界時）になっていることが多く、日本時間（JST, UTC+9）を期待していると、表示される時刻が9時間早く見えることがある。

対処法としては、実行前に環境変数 `TZ` を設定する方法がある。

```
export TZ=Asia/Tokyo
./diary_v4_1
```

もしくは、プログラム内で明示的にタイムゾーンを設定する方法もある（Linux/macOS環境）。

```c
setenv("TZ", "Asia/Tokyo", 1);
tzset();
```

なお、この対処は後続のVer.6で正式に取り込まれている（`init_locale_utf8()` 導入と合わせてロケール・タイムゾーンまわりを堅牢化）。Ver.4.1の時点ではまだ導入していない。
